## 1. 阅读指引（必读）
- `GlobalUser/AGENTS.md`、`GlobalUser/CLAUDE.md`、`GlobalUser/GEMINI.md` 为全局用户级；仓库根目录同名文件为项目级。
- 本文件仅承载跨项目稳定规则与 Gemini 平台差异，不放仓库私有实现细节。
- 三层固定：共性基线（A）+ 平台差异（B）+ 项目差异（C）；D 为维护附录。
- 单文件必须自包含：可独立理解适用范围、硬约束、执行动作、风险与回滚。
- 冲突处理：平台加载语义 + 就近原则 + 项目优先，并在回复中明确采用依据。

## A. 共性基线（全局）
### A.1 语言与输出
- 默认中文沟通；代码、命令、日志、错误原文保持英文。
- 简单任务使用 `Result + Evidence`；复杂任务使用 `Goal / Plan / Changes / Verification / Risks`。
- 默认仅单条最终回复；仅在真实阻塞或不可逆高风险前允许额外消息。

### A.2 终态方案先行闸门（新建/扩展/修复/重构必做）
- 触发场景：新建项目、新建子系统、扩展功能、新增能力、性能优化、错误修复、重构。
- 动手前必须先完成“需求边界 -> 技术栈 -> 目标架构 -> 分阶段落地”方案，先定终态再定实现。
- 需求边界最少包含：核心功能、非功能指标（性能/稳定性/安全性）、上下游依赖、兼容约束、明确 out-of-scope。
- 终态目标必须对齐最佳实践：模块化、结构化、规范化、长期可维护。
- 方案最少包含：模块边界、依赖方向、状态归属、数据边界、验证策略、迁移步骤、回滚路径。
- 若现状与终态不一致，必须同时写清：当前落点、目标归宿、过渡接线方式、阶段性回滚点。

### A.3 归宿判定与反堆叠闸门（扩展/修复必做）
- 扩展、新增、修复前，先判定归宿层与模块边界，再落代码或规则。
- 禁止把业务规则、状态写回、外部交互分支、容错补丁继续堆入热点入口、窗口文件、工具类或“万能服务”。
- 命中热点区时优先抽离后接线；至少不得新增同类结构债。
- 紧急止血允许先做，但必须同步说明最终归宿，不得把临时方案沉淀为长期结构。

### A.4 变更与安全边界
- 仅修改需求相关内容；无授权不得改公共 API、Schema、序列化格式或外部行为。
- 新增/升级依赖前，先说明必要性、影响面、验证方式与回退路径。
- 高风险操作（如 `rm -rf`、`git reset --hard`、批量覆盖、破坏性数据改写）前必须提示影响。
- 不输出密钥、令牌或隐私数据；必要时脱敏。

### A.5 验证与回滚
- 改动后执行最小相关验证；无法验证时必须写明原因、风险、建议补测与回滚点。
- 批量改动（`>=2` 文件或跨模块）必须说明影响范围、关键风险与回滚动作。
- 验证结论必须可复现，避免只给主观判断。

### A.6 全局-项目协同接口矩阵（1+1>2）
- 仅改全局：跨项目稳定规则、平台加载/配置/诊断语义、统一评分口径、维护同步策略。
- 仅改项目：仓库结构、领域边界、构建测试发布、热点区限制、数据与隐私约束、降级策略。
- 两层联动：项目实践暴露全局模板缺口时，先改 GlobalUser，再回写项目最小复述与仓库动作。
- 协同交付链必须闭环：`规则判定 -> 仓库落点 -> 执行命令 -> 验证证据 -> 回滚动作`。

### A.7 自包含与精简度控制
- 禁止过度冗余：同一规则只定义一次，不把 B 节平台差异复制到 A/C/D。
- 禁止过度精简：不能只留原则口号，必须保留可执行动作、边界判定与回滚提示。
- 每次改写后至少复核：边界清晰度、执行性、重复项、缺失项、精简度。

### A.8 评估口径
- 评估类任务统一给出 0-100 分、主要问题、可执行改进动作。
- 协同评分五维固定：覆盖度、边界清晰、执行性、冗余控制、跨平台一致性（各 20 分）。
- 任一文件低于 80 分，必须给出明确补救动作与责任归属。

## B. Gemini 平台差异（全局，基于官方文档）
### B.1 加载与作用域
- 默认规则文件名为 `GEMINI.md`，全局/项目/子目录规则分层叠加。
- `.geminiignore` 与 `.gitignore` 可能影响上下文发现范围；规则未生效时先检查目录与 ignore。
- 若临时调整规则文件名或发现范围，任务结束后必须回退并复测。

### B.2 配置与上限
- context discovery、规则发现上限、memory 相关字段以官方文档为准，不在本文件硬编码。
- 配置试验前后记录：变更参数、目的、生效证据、回滚动作。
- 临时扩大扫描范围的配置，结论确认后必须恢复基线。

### B.3 诊断与命令
- 通过 memory 或等效命令核对当前加载上下文、目录范围与最近注入内容。
- 若发现上下文偏差，优先刷新上下文并排查目录/ignore，再决定是否改规则文件。
- 排查新增的临时记忆、临时上下文说明，任务结束后必须清理。

### B.4 时效性提示
- 若本文件与当前官方文档冲突，以官方文档为准，并在回复中标注差异。

## C. 项目差异（全局模板）
- 项目级文件必须完整自包含，并显式指向对应 `GlobalUser/同名文件`。
- 项目级需把全局硬约束落成仓库动作：项目概况、模块边界、数据与隐私、构建测试发布、风险降级、验证回滚。
- 项目级 A/C/D 承担仓库共性与领域规则；B 节只写该平台在本仓库的加载/配置/诊断差异。
- 项目级必须提供“仅项目 / 仅全局 / 两层联动”判定矩阵，防止职责重叠或缺失。
- 项目级涉及目标架构时，必须同时给出当前仓库落点与迁移路径。
- 项目级不得复制全局 B 节平台差异正文，只保留本仓库所需最小差异。

## D. 维护校验清单（全局）
说明：维护清单为附录，不属于三层结构正文。
- 保持 `1 / A / B / C / D` 结构完整且编号一致。
- GlobalUser 三文件 A/C/D 必须一致，仅 B 节允许平台差异。
- 项目三文件 A/C/D 必须一致，仅 B 节允许平台差异。
- 六文件引用链必须一致：GlobalUser 对应映射到项目同名文件，语义无冲突。
- 复核三层边界：全局不吞项目，项目不复写平台，合并后无空白职责。
- 复核执行闸门：终态方案先行、反堆叠执行、验证回滚、边界矩阵均可直接执行。
- 复核协同效能：全局与项目形成闭环链路（规则 -> 落点 -> 验证 -> 回滚），不得断链。
- 复核精简度：删重复、删口号、补动作、补风险、补回滚。
- 标题版本号、`**版本**`、日期、变更记录必须同步。
